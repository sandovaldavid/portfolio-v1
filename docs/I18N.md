# Internationalization architecture

This document is the current operating contract for English and Spanish content in the portfolio. It describes implemented ownership and the temporary compatibility boundary that remains under issue #143; migration history and rejected alternatives belong in Cortex-L7.

## Current status

- **Implemented:** Astro-native locale detection and URL generation, granular typed UI catalogs, localized profile/experience/project Content Collections, editorial translation identities, localized metadata and executable source/route/browser checks.
- **In progress:** issue #143 removes the remaining legacy dictionaries, mixed-value translator, `useTranslationsList()` consumers and six baselined legacy routes.
- **Deprecated:** those legacy paths may remain only for existing consumers until #143 completes. They must not receive new domains, keys or exceptions.

See [STATUS.md](STATUS.md) for the repository-wide classification.

## Locale and route contract

The supported locales are:

- `en`: default locale, rendered without a URL prefix;
- `es`: Spanish locale, rendered below `/es`.

Astro's built-in i18n configuration is the canonical routing contract. `Astro.currentLocale` owns locale detection and `astro:i18n` owns ordinary localized URL generation. The default locale is a routing default only; it is not a fallback for missing Spanish content.

## Source ownership

Every localized value has exactly one owner.

| Content category | Canonical source | Examples |
| --- | --- | --- |
| Locale configuration and routing | `astro.config.mjs` and typed shared configuration | supported locales, default locale, prefix policy |
| Small reusable UI copy | `src/shared/config/i18n/locales/{locale}/` | buttons, navigation, labels, validation messages, ARIA text |
| Structured portfolio content | schema-validated Astro Content Collections | profile, biography, experience, projects, case studies |
| Editorial content | localized Astro Content Collections | blog posts and devlog entries |
| Language-neutral technical data | TypeScript records keyed by stable IDs | URLs, assets, technology IDs, ordering, feature flags |
| Component behavior and presentation | Astro or TypeScript components | markup, state, interactions, layout and styling |

A component must not become a translation store. Local bilingual `copy` objects, locale ternaries containing complete messages and hardcoded user-facing strings are prohibited for new work.

## Granular UI catalogs

UI catalogs contain short scalar strings only. They must not contain experience records, project narratives, biography paragraphs or other structured domain content.

Current mirrored module roots include:

```text
src/shared/config/i18n/locales/
├── en/
│   ├── accessibility.json
│   ├── breadcrumbs.json
│   ├── cli.json
│   ├── common.json
│   ├── errors.json
│   ├── footer.json
│   ├── metadata.json
│   ├── navigation.json
│   ├── recruiter.json
│   ├── splash.json
│   ├── theme.json
│   └── sections/
│       ├── about.json
│       ├── badges.json
│       ├── experience.json
│       ├── hero.json
│       ├── projects.json
│       ├── research.json
│       ├── tech-stack.json
│       └── vision.json
└── es/
    └── the same module paths and keys
```

Rules:

- use kebab-case file names and camelCase semantic leaf keys;
- derive namespaces from relative module paths;
- keep identical module paths and key sets in English and Spanish;
- keep values non-empty scalar strings without HTML;
- translate complete grammatical units instead of concatenating fragments;
- split a module when it gains more than one cohesive responsibility;
- do not add catch-all replacements such as `home.json`, `pages.json`, `misc.json` or one new monolithic file per locale.

The exact catalog API and addition workflow are documented in [I18N-CATALOGS.md](I18N-CATALOGS.md).

## Structured portfolio content

Profile, biography, experience and projects use focused Content Collections:

```text
src/content/
├── portfolio-profile/{locale}/profile.json
├── experience/{locale}/<experience-id>.json
└── projects/{locale}/<project-id>.json
```

Every localized structured entry includes or derives:

- `locale`: `en` or `es`;
- an immutable language-neutral ID such as `profileId`, `experienceId` or `projectId`;
- localized fields owned by the entry;
- a route slug when the public URL is not derived from the stable identity.

Language-neutral metadata remains in TypeScript and joins localized content by stable ID. Do not duplicate repository URLs, demo URLs, image imports, technology IDs, ordering or feature flags per locale unless the value is genuinely locale-specific.

## Blog and devlog

Blog and devlog are localized editorial Content Collections. Every entry requires an immutable language-neutral `translationKey` that identifies the same editorial work across locales.

A localized slug may differ between languages. Translation pairing uses `translationKey`, never filename equality, slug equality or string replacement.

Published entries should normally have English and Spanish counterparts, but explicitly documented single-language editorial entries are allowed. When a counterpart is unavailable:

- the detail page passes only verified counterpart routes to the language picker;
- the unavailable locale is disabled instead of linking to a non-existent route;
- a draft is not a production counterpart;
- no automatic fallback changes the language of the current page.

RSS remains blog-only and locale-specific: `/rss.xml` contains published English posts and `/es/rss.xml` contains published Spanish posts.

## Locale resolution and links

- Read `Astro.currentLocale` in Astro routes, layouts and components.
- Convert the framework value through `getLanguageFromLocale()` only when the typed `Language` enum is required.
- Generate ordinary localized links with `getRelativeLocaleUrl()` from `astro:i18n`.
- `Layout.astro` derives the locale-neutral current route and passes a verified route map to the language picker.
- Blog and devlog detail routes replace ordinary route mapping with counterparts resolved by `translationKey`.
- The language picker preserves query strings and fragments but does not parse or rewrite locale prefixes.
- English URLs never receive an `/en` prefix while `prefixDefaultLocale` is `false`.

The removed `getLangFromUrl()` and `getLocalizedPath()` helpers must not be reintroduced. A custom helper is acceptable only for domain behavior Astro does not provide, such as resolving an existing translated content entry.

## Missing translations and fallback

Missing translations are build defects.

- catalog module/key parity and localized stable-ID parity are checked;
- empty required values fail validation;
- catalog resolution fails with the locale and key;
- generated routes and browser tests verify language, canonical, alternate and picker targets;
- production must not silently emit a key name or English fallback on a Spanish page;
- `DEFAULT_LANGUAGE` resolves an undefined or unsupported framework locale only, not missing localized content.

## Rich text and HTML safety

UI catalog values are plain text and must not be rendered through `set:html`.

Use Markdown/MDX or structured content for long-form and formatted copy. Allow only repository-owned MDX components with reviewed props. Treat external or user-provided content as untrusted and never inject it as translation markup.

## Accessibility, SEO and metadata

Localization includes visible, assistive and machine-readable text:

- accessible names and screen-reader-only status text;
- image alternative text and figure descriptions;
- page titles and descriptions;
- Open Graph and Twitter metadata;
- breadcrumb names;
- RSS titles and descriptions;
- language-dependent JSON-LD values.

Brand names, company names, technology names and source-code identifiers remain untranslated when that preserves official identity. See [I18N-METADATA.md](I18N-METADATA.md).

## Temporary compatibility boundary

Until issue #143 is complete, the following **Deprecated** paths may remain only for unmigrated consumers:

- root legacy `en.json` and `es.json` dictionaries;
- the old flattening and mixed-value translation implementation;
- `useTranslationsList()`;
- the six exact legacy route files covered by `HARD_CODED_COPY_DEBT_BASELINE`.

Rules for this period:

- do not add new domains, keys, consumers or debt files;
- necessary fixes must update both locales within the existing owner;
- do not broaden allowlists or replace exact debt hashes with directory exclusions;
- remove compatibility code only together with its final consumers and associated tests;
- no parallel i18n runtime may be introduced without an explicit new decision.

The executable details are in [I18N-ENFORCEMENT.md](I18N-ENFORCEMENT.md).

## Contributor checklist

Before adding or changing user-facing text:

1. classify it as UI copy, structured portfolio content, editorial content or language-neutral data;
2. change only the canonical owner;
3. add English and Spanish values together unless an editorial entry is explicitly single-language;
4. use stable IDs for structured content and `translationKey` for editorial content;
5. include accessibility and metadata text in the same change;
6. add or update the relevant parity, schema, route or browser regression;
7. run the repository's required local gates.

```bash
bun run check
bun run test:unit:ci
bun run build
bun run check:links
bun run test:e2e:smoke
```

## Prohibited patterns

Do not add:

- one catch-all locale file per language;
- arrays or domain records in the generic UI catalog;
- bilingual copy maps inside production components;
- visible or accessible text selected with locale ternaries;
- hardcoded English UI text on bilingual routes;
- raw HTML inside translation catalogs;
- automatic English fallback that hides incomplete Spanish content;
- manual locale-prefix concatenation for ordinary routing;
- route switching that constructs a content target without verifying it exists;
- duplicated language-neutral technical metadata in every locale.
