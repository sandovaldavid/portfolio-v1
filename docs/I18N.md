# Internationalization architecture

This document is the active operating contract for English and Spanish content in the portfolio. The architectural decision and rejected alternatives are recorded in [ADR-0001](adr/0001-granular-bilingual-content-architecture.md).

## Locale and route contract

- `en` is the default locale and has no URL prefix.
- `es` is served below `/es`.
- `Astro.currentLocale` owns locale detection.
- `astro:i18n` owns ordinary localized URL generation.
- The default locale is a routing default, not a fallback for missing Spanish content.

Use `getLanguageFromLocale()` only to convert Astro's locale into the typed `Language` enum. Use `getRelativeLocaleUrl()`, `getPathByLocale()` and `pathHasLocale()` for ordinary routes. Editorial detail routes may replace ordinary paths only with verified counterparts resolved by stable translation identity.

The removed `getLangFromUrl()` and `getLocalizedPath()` helpers must not be reintroduced.

## Canonical ownership

Every localized value has exactly one owner.

| Category | Canonical source | Examples |
|---|---|---|
| Locale configuration and routing | `astro.config.mjs` and typed shared configuration | supported locales, default locale, prefix policy |
| Small reusable UI copy | mirrored granular catalogs under `src/shared/config/i18n/locales/{locale}/` | controls, navigation, labels, metadata, ARIA text |
| Structured portfolio content | schema-validated Content Collections | profile, experience, research, projects, case studies |
| Editorial content | localized Markdown/MDX Content Collections | blog posts and devlog entries |
| Language-neutral technical data | TypeScript records keyed by stable IDs | URLs, assets, technology IDs, ordering, feature flags |
| Behavior and presentation | Astro and TypeScript components | markup, state, interactions and styling |

A component must not become a translation store. Production code must not contain local bilingual copy maps, complete-message locale ternaries, direct locale-file imports or hardcoded user-facing phrases.

## Granular scalar catalogs

Catalog values are non-empty plain scalar strings. Arrays, records, biography paragraphs, project narratives and research datasets belong in Content Collections.

```text
src/shared/config/i18n/
├── catalog.ts
├── languages.ts
├── types.ts
└── locales/
    ├── en/
    │   ├── accessibility.json
    │   ├── breadcrumbs.json
    │   ├── cli.json
    │   ├── common.json
    │   ├── errors.json
    │   ├── footer.json
    │   ├── metadata.json
    │   ├── navigation.json
    │   ├── recruiter.json
    │   ├── splash.json
    │   ├── theme.json
    │   ├── pages/
    │   │   ├── atena.json
    │   │   ├── blog.json
    │   │   ├── components.json
    │   │   ├── devlog.json
    │   │   ├── research.json
    │   │   └── skills.json
    │   └── sections/
    │       ├── about.json
    │       ├── badges.json
    │       ├── experience.json
    │       ├── hero.json
    │       ├── projects.json
    │       ├── research.json
    │       ├── tech-stack.json
    │       └── vision.json
    └── es/
        └── ... identical module paths and key sets
```

Rules:

- use kebab-case file names and camelCase semantic leaf keys;
- derive namespaces from relative paths, such as `pages.blog` or `sections.hero`;
- keep English and Spanish module paths and keys in exact parity;
- store complete grammatical units rather than translated fragments;
- register every module in `catalog.ts` and `UI_CATALOG_MODULES`;
- consume copy through `createScopedUiTranslator()` or the other typed catalog APIs;
- never recreate catch-all `en.json` or `es.json` files.

Missing or empty catalog values throw `MissingUiTranslationError`; production never returns a key name or silently falls back to English.

## Structured localized content

```text
src/content/
├── portfolio-profile/{en,es}/profile.json
├── experience/{en,es}/<experience-id>.json
├── research/{en,es}/<research-id>.json
└── projects/{en,es}/<project-id>.json
```

Each entry declares its locale and an immutable language-neutral identity:

- `profileId` for portfolio profile content;
- `experienceId` for professional roles;
- `researchId` for research records;
- `projectId` for projects and case studies.

Entities load and validate these collections through public APIs. Components consume entities rather than `astro:content` directly. Language-neutral metadata remains in TypeScript and joins localized records by stable ID.

Research narrative, metrics, pipeline steps, architecture and engineered features belong to the research collection. Only page labels and metadata belong to `pages.research`.

## Blog and devlog

Blog and devlog remain localized editorial Content Collections. Every entry has an immutable `translationKey` that pairs the same work across locales independently of filename or slug.

- indexes exist in both locales;
- detail routes expose only verified counterparts;
- missing or draft-only counterparts remain unavailable;
- no automatic fallback changes the language of the current page;
- `/rss.xml` contains published English blog posts;
- `/es/rss.xml` contains published Spanish blog posts.

Page-level labels and metadata use `pages.blog` and `pages.devlog`; article bodies remain Markdown or MDX.

## Rich text and HTML safety

Catalogs contain plain text and must not contain HTML or use `set:html`.

Use Markdown/MDX for long-form formatted content or a typed structured schema for repeatable blocks. Keep complete sentences together and never build translated grammar from fragments around markup. External or user-provided values are untrusted and must not become translation markup.

## Accessibility, SEO and metadata

The localization contract includes:

- accessible names, descriptions and screen-reader-only copy;
- image alternative text;
- page titles and descriptions;
- Open Graph and Twitter metadata;
- breadcrumbs;
- RSS titles and descriptions;
- language-dependent JSON-LD values.

Official brand, company and technology names remain language-neutral when appropriate. Ambiguous exceptions require an exact documented allowlist entry.

## Enforcement

The repository enforces the final architecture with:

```bash
bun run check:i18n:catalogs
bun run check:i18n:content
bun run check:i18n:hardcoded
bun run check:i18n
bun run build
bun run check:i18n:routes
bun run test:e2e:smoke
```

The checks reject catalog drift, empty values, missing structured counterparts, unsupported locales, hardcoded visible or accessible copy, local bilingual maps, complete-message locale ternaries, broken route targets and known English-only phrases on Spanish output.

There is no migration-debt baseline or compatibility bypass. The monolithic dictionaries, flattened `string | string[]` runtime, `useTranslations()` and `useTranslationsList()` have been removed.

See [I18N-CATALOGS.md](I18N-CATALOGS.md) for the catalog API and [I18N-ENFORCEMENT.md](I18N-ENFORCEMENT.md) for diagnostics and contributor workflow.

## Contributor checklist

Before changing user-facing text:

1. classify it as UI copy, structured content, editorial content or language-neutral data;
2. update only the canonical owner;
3. update English and Spanish together unless a single-language editorial exception is explicitly documented;
4. preserve stable IDs and editorial `translationKey` values;
5. include accessibility and metadata impact;
6. add or update the relevant schema, parity, route or browser regression;
7. run the repository gates described in [TESTING.md](TESTING.md).

## Prohibited patterns

Do not add:

- monolithic locale files;
- a second translation runtime;
- arrays or domain records in the scalar catalog;
- direct imports of locale JSON outside `catalog.ts`;
- component-local bilingual copy objects;
- hardcoded bilingual route copy;
- locale ternaries selecting complete messages;
- raw HTML in catalog strings;
- silent English fallback;
- manual locale-prefix parsing or concatenation for ordinary routes;
- unverified editorial route switching;
- duplicated language-neutral metadata in each locale.
