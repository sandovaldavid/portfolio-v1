# Internationalization architecture

This document is the active operating contract for English and Spanish content in the portfolio. The architectural decision and rejected alternatives are recorded in [ADR-0001](adr/0001-granular-bilingual-content-architecture.md).

## Scope and route contract

The supported locales are:

- `en`: default locale, rendered without a URL prefix;
- `es`: Spanish locale, rendered below `/es`.

Astro's built-in i18n configuration is the canonical routing contract. `Astro.currentLocale` owns locale detection and `astro:i18n` owns ordinary localized URL generation. The default locale is a routing default only; it is not a fallback for missing Spanish content.

## Source ownership

Every localized value has exactly one owner.

| Content category                    | Canonical source                                                          | Examples                                                        |
| ----------------------------------- | ------------------------------------------------------------------------- | --------------------------------------------------------------- |
| Locale configuration and routing    | `astro.config.mjs` and typed shared configuration                         | supported locales, default locale, route prefix policy          |
| Small reusable UI copy              | granular locale catalogs under `src/shared/config/i18n/locales/{locale}/` | buttons, navigation, labels, validation messages, ARIA text     |
| Structured portfolio content        | schema-validated Astro Content Collections                                | profile, biography, experience, projects, case studies          |
| Editorial content                   | localized Astro Content Collections                                       | blog posts and devlog entries                                   |
| Language-neutral technical data     | TypeScript records keyed by stable IDs                                    | URLs, image references, technology IDs, ordering, feature flags |
| Component behavior and presentation | Astro or TypeScript components                                            | markup, state, interactions, layout and styling                 |

A component must not become a translation store. Local `copy` objects, locale ternaries containing complete messages and hardcoded user-facing strings are transitional debt, not accepted patterns for new work.

## Granular UI catalogs

UI catalogs contain short scalar strings only. They must not contain arrays of experience records, project case studies, biography paragraphs or other domain content.

The target structure is mirrored between locales:

```text
src/shared/config/i18n/
├── config.ts
├── types.ts
├── catalog.ts
└── locales/
    ├── en/
    │   ├── common.json
    │   ├── navigation.json
    │   ├── accessibility.json
    │   ├── theme.json
    │   ├── errors.json
    │   ├── recruiter.json
    │   ├── splash.json
    │   └── sections/
    │       ├── hero.json
    │       ├── about.json
    │       ├── experience.json
    │       ├── projects.json
    │       ├── research.json
    │       ├── vision.json
    │       └── tech-stack.json
    └── es/
        └── ... the same module paths and keys
```

This is a required boundary, not a fixed final list. Split a module when it gains more than one cohesive responsibility. Do not create catch-all files such as `home.json`, `pages.json`, `misc.json` or a replacement monolithic `en.json` and `es.json`.

### Catalog naming

- Use kebab-case for file names: `tech-stack.json`.
- Derive the namespace from the relative file path: `sections.hero`.
- Use camelCase semantic leaf keys: `sections.hero.viewWork`.
- Name keys after stable intent, not the current sentence: use `navigation.resume`, not `navigation.downloadMyLatestResumePdf`.
- Keep identical module paths and key sets in `en` and `es`.
- Keep catalog values scalar strings. Structured arrays and records belong in content schemas.
- Do not concatenate translated fragments. A translation must own a complete grammatical unit.

The English catalog provides the compile-time key shape. Spanish must satisfy the same module and key structure without being forced to reuse English literal values.

## Structured portfolio content

Profile, biography, experience and projects are content, not UI labels. They use focused Content Collections:

```text
src/content/
├── portfolio-profile/
│   ├── en/
│   │   └── profile.json
│   └── es/
│       └── profile.json
├── experience/
│   ├── en/
│   │   └── <experience-id>.json
│   └── es/
│       └── <experience-id>.json
└── projects/
    ├── en/
    │   └── <project-id>.json
    └── es/
        └── <project-id>.json
```

Large profile content may be split into focused entries such as `summary`, `biography` and `focus` when that improves ownership. Experience and projects use one entry per stable identity and locale.

Every localized structured entry must include or derive:

- `locale`: `en` or `es`;
- an immutable language-neutral domain ID such as `profileId`, `experienceId` or `projectId`;
- localized fields owned by the entry;
- a route slug when the public URL is not derived from the stable identity.

Examples of stable identities are `atena-software-engineer` and `yukidoke`. Changing displayed titles or localized slugs must not change the domain ID.

Language-neutral metadata remains in TypeScript and joins localized content by its stable domain ID. Do not duplicate repository URLs, demo URLs, image imports, technology IDs, ordering or feature flags in every locale unless the value is genuinely locale-specific.

Project entries own localized title, description, category, image alternative text, case-study narrative, duration, role and optional evidence copy. The project metadata record owns slugs, URLs, assets, technology IDs, ordering, featured state and evidence-source URLs. The entity must reject duplicate project IDs, missing locale counterparts and localized evidence sources without technical metadata.

## Blog and devlog

Blog and devlog are localized editorial Content Collections. Every entry requires an immutable, language-neutral `translationKey` that identifies the same editorial work across locales.

A localized slug may differ between languages. Translation pairing therefore uses `translationKey`, never filename equality, slug equality or string replacement. Each collection rejects duplicate identities within the same locale before returning editorial content.

Published blog posts and devlog entries should normally have English and Spanish counterparts, but intentionally single-language editorial entries are allowed. Missing counterparts follow this behavior:

- detail pages pass only verified counterpart routes to the language picker;
- an unavailable target locale is rendered as disabled instead of linking to a non-existent route;
- a blog draft is unavailable as a production counterpart even when it shares the same `translationKey`;
- collection indexes remain available in both supported locales;
- no automatic content fallback changes the language of the current page.

RSS remains blog-only and locale-specific. `/rss.xml` contains published English posts, `/es/rss.xml` contains published Spanish posts, and draft filtering is unchanged.

## Rich text and HTML safety

UI catalog values are plain text. They must not contain HTML and must not be rendered through `set:html`.

Use these rules for formatted content:

1. Store long-form or markup-bearing copy as Markdown or MDX in a Content Collection.
2. Allow only repository-owned MDX components with reviewed props and no unsafe raw HTML path.
3. Use structured content blocks when a schema is more appropriate than MDX.
4. Keep complete sentences together; do not assemble grammar from translated fragments around `<strong>` elements.
5. Treat external or user-provided content as untrusted and never inject it as translation markup.

When a short UI sentence only needs visual emphasis, prefer styling the complete localized string or move the formatted copy to structured content. Do not add HTML to a catalog as a shortcut.

## Missing translations and fallback behavior

Missing translations are defects.

The target behavior is:

- module or key parity is checked before build;
- empty required values fail validation;
- development and test resolution fails loudly with the locale, namespace and key;
- production builds fail rather than silently shipping a key name or English fallback on a Spanish page;
- `DEFAULT_LANGUAGE` applies only when resolving an undefined or unsupported framework locale, not when localized content is absent.

The current key-returning behavior in `useTranslations()` is transitional and must not receive new domains. Automated catalog, content and route checks are completed by issue #141.

## Locale and route resolution

Astro owns locale detection and ordinary localized URL generation.

- Read `Astro.currentLocale` in Astro routes, layouts and components.
- Convert the framework value through `getLanguageFromLocale()` only when the typed `Language` enum is required. This adapter does not inspect URLs or generate paths.
- Generate ordinary localized links with `getRelativeLocaleUrl()` from `astro:i18n`.
- `Layout.astro` uses `pathHasLocale()` and `getPathByLocale()` to derive the locale-neutral current route, then creates the verified English/Spanish route map consumed by the language picker.
- Blog and devlog detail routes replace that ordinary map with counterparts resolved by `translationKey`; a missing counterpart remains unavailable.
- The language picker never parses or rewrites locale prefixes. Its client script only preserves the current query string and fragment across initial load, hash changes and `ClientRouter` navigation.
- Explicit trailing slashes are retained when deriving the locale-neutral route.
- English URLs never receive an `/en` prefix while `prefixDefaultLocale` remains `false`.

The removed `getLangFromUrl()` and `getLocalizedPath()` helpers must not be reintroduced. A custom routing helper is acceptable only when it adds domain behavior Astro does not provide, such as resolving an existing translated content entry.

## Accessibility, SEO and metadata

Localization includes non-visible and machine-readable text:

- `aria-label`, `aria-labelledby` support text and screen-reader-only copy;
- image alternative text and figure descriptions;
- page titles and descriptions;
- Open Graph and Twitter metadata;
- breadcrumb names;
- RSS titles and descriptions;
- language-dependent JSON-LD values.

Brand names, company names, technology names and source-code identifiers remain untranslated when that preserves their official identity. Document ambiguous exceptions near the owning content schema or catalog.

## Migration policy

The migration is incremental and follows roadmap issue #144:

1. #142 upgrades Astro and verifies the supported framework APIs.
2. #132 introduces typed granular UI catalogs and parity validation.
3. #133 and #134 migrate shell and home-page UI copy.
4. #135, #136 and #137 migrate profile, experience and projects to content collections.
5. #138 implements explicit editorial pairing; #139 implements Astro-native locale detection and URL generation.
6. #140 and #141 complete metadata, accessibility and executable enforcement.
7. #143 removes the remaining legacy dictionaries and mixed-value translation API.

Until #143 is complete:

- the legacy `en.json`, `es.json`, `flattenJSON`, `useTranslationsList()` and component-local copy maps may remain only for unmigrated consumers;
- do not extend those patterns to new domains;
- necessary bug fixes must update both locales and stay within the current owner until the relevant migration issue lands;
- do not delete compatibility code before every consumer has migrated and the replacement checks are active;
- do not introduce i18next or another parallel runtime without a new accepted ADR.

Project narratives no longer use the legacy dictionaries or the project-specific Yukidoke merge. New project and case-study copy must be added only through paired entries in `src/content/projects/{locale}/`.

## Contributor checklist

Before adding or changing user-facing text:

1. Classify it as UI copy, structured portfolio content, editorial content or language-neutral data.
2. Change the canonical owner only; do not duplicate the same fact in components and catalogs.
3. Add English and Spanish values together unless an editorial entry is deliberately documented as single-language.
4. Use the correct stable domain ID for structured portfolio content and `translationKey` for editorial content.
5. Include accessibility and metadata text in the same change.
6. Add or update the parity, schema, route or browser regression required by the owning issue.
7. Run the repository's required local gates.

For documentation-only changes:

```bash
bun install --frozen-lockfile
bun run format:check
bun run check:docs
```

For implementation changes, also run the canonical and change-specific commands defined in [AGENTS.md](../AGENTS.md) and [TESTING.md](TESTING.md).

## Prohibited patterns

Do not add:

- a single catch-all locale file per language;
- arrays or domain records in the generic UI translator;
- bilingual `copy` maps inside production components;
- visible or accessible text selected with `locale === 'en' ? ... : ...`;
- hardcoded English UI text on bilingual routes;
- raw HTML inside translation catalogs;
- automatic English fallback that hides incomplete Spanish content;
- regular expressions or manual locale-prefix concatenation for ordinary routing;
- route switching that constructs a content target without verifying it exists;
- duplicated technical metadata in every locale.
